# 方案定稿 (Design Gate)

对 **$ARGUMENTS** 进行架构设计，经产品视角挑战后判定，通过则写入设计文档。

## 三阶段工作流

### 阶段 1: 架构设计

#### 1. 上下文收集（自身完成）

在调用 agent 之前，先收集相关上下文:

- 阅读 `docs/system_design.md` 中与 $ARGUMENTS 相关的章节
- 用 Grep/Glob 查找项目中与 $ARGUMENTS 直接相关的代码文件
- 整理出: 涉及哪些模型/表、哪些 API 路由、哪些服务模块

#### 2. 启动 architect-designer agent（使用 Agent tool）

prompt 应包含:
- 架构任务: $ARGUMENTS
- **附上步骤 1 收集到的上下文**（相关文件路径、现有数据模型、当前实现概要）
- 要求输出结构化技术方案: 背景与目标、数据模型、API 设计、关键流程、影响范围、迁移计划、替代方案

### 阶段 2: 产品挑战

启动 `product-reviewer` agent（**挑战者立场**）:

prompt 应包含:
- 阶段 1 的技术方案全文
- 要求以挑战者立场审查:
  - 技术方案是否满足用户场景？
  - 有没有过度设计或遗漏场景？
  - 投入产出比合理吗？
- 要求输出结构化评审报告

### 阶段 3: 判定 + 自动动作

#### 判定标准

- **✅ 通过**: 无架构级红色问题 + 产品确认覆盖核心场景
- **⚠️ 有条件通过**: 方案可行但有待优化点
- **❌ 不通过**: 架构问题或严重场景遗漏

#### 通过动作

将技术方案**追加/更新**到 `docs/system_design.md` 对应章节（使用 Edit tool）。

#### 不通过动作

输出需重新设计的部分，要求用户决策。

#### 输出格式

```
## 方案定稿: {任务描述}

### 技术方案（来自 architect-designer）
{数据模型、API 设计、关键流程、影响范围、迁移计划}

### 产品挑战（来自 product-reviewer）
{场景覆盖度、过度设计风险、投入产出比}

### 判定: ✅ 通过 / ⚠️ 有条件通过 / ❌ 不通过
{判定理由}

### 待优化项（如有）
{有条件通过时需后续处理的点}

### 后续建议
{可执行的下一步行动}
```
