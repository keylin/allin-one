# 统一验收 (Acceptance)

对 **$ARGUMENTS** 进行一站式全面验收。如果未提供参数，验收最近的 git 变更。

## 工作流程

### 阶段 1: 产品验收 + 功能测试

> **代码质量已由 `/g-develop` 阶段 2 保证**（code-reviewer + code-tester），此处不再重复。

**启动 `product-reviewer` agent** — 产品体验：功能完整性、用户体验、视觉布局、可发现性

agent 的 prompt 应包含:
- 验收目标: $ARGUMENTS（如果为空则说明"验收最近的 git 变更，先运行 git diff --name-only 确定范围"）
- 要求输出结构化报告（按 agent 定义的输出格式）

**同时自身执行功能路径测试**:

阅读 `docs/product_spec.md` 和 `docs/system_design.md`，对变更范围进行:

- **正常路径**: 核心 CRUD 流程是否闭环，端到端数据流是否通畅
- **边界条件**: 空数据、极端值、首次使用、分页边界
- **异常路径**: 无效 ID、配置缺失、格式错误
- **回归检查**: 变更是否影响关联功能，配置/环境变量是否需更新

### 阶段 2: 文档同步（阶段 1 通过后）

仅当阶段 1 **无 🔴 问题**时才执行。如果阶段 1 有阻塞问题，跳过此阶段直接输出报告。

启动 `doc-maintainer` agent — 评估代码变更对项目文档的影响。

### 阶段 3: 合并输出

```
## 验收报告: {功能/变更描述}

### 验收结论: ✅ 通过 / ⚠️ 有条件通过 / ❌ 不通过

### 产品体验（来自 product-reviewer）
{功能完整性、用户体验、视觉布局评估}

### 功能测试（自身执行）
{正常路径、边界条件、异常路径、回归检查}

### 文档影响（来自 doc-maintainer）
{需要更新的文档及建议}

### 问题清单（合并去重，按优先级排序）
#### 🔴 必须修复（阻塞上线）
#### 🟡 建议修复（不阻塞但应尽快处理）
#### 🔵 可优化（锦上添花）

### 亮点
{做得好的地方}
```

### 验收结论判定标准

- **✅ 通过**: 无 🔴 问题，🟡 问题不超过 2 个
- **⚠️ 有条件通过**: 无 🔴 问题，但有多个 🟡 问题需后续跟进
- **❌ 不通过**: 存在 🔴 问题，必须修复后重新验收

### 修复后复查

如果验收不通过，修复后重新跑 `/g-accept` 验收。

### 阶段 4: 自动提交（验收通过后）

当验收结论为 **✅ 通过** 或 **⚠️ 有条件通过** 时，自动执行 git commit:

1. `git add` 暂存所有相关变更文件（不使用 `git add -A`，逐文件添加）
2. commit message 格式:
   ```
   feat/fix/refactor: {变更描述}

   验收通过: {通过/有条件通过}
   - 产品体验: ✅
   - 功能测试: ✅
   - 文档同步: ✅ / N/A
   ```
3. 如果 doc-maintainer 产生了文档更新，一并包含在同一个 commit 中
4. **不自动 push**，仅本地 commit

如果验收 **❌ 不通过**，不执行 commit，在报告末尾说明"修复后重新验收"。
